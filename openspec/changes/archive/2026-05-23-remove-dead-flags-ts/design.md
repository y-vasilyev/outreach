## Context

Чисто-удалительный change. Удаляется один файл (`packages/shared/src/flags.ts`) и точечные упоминания в `packages/shared/src/index.ts` (re-export), `packages/shared/src/feature-flags.ts` (docstring), `packages/shared/package.json` (subpath export `./flags`), `CLAUDE.md` (раздел про фичефлаги), `DESIGN.md` (то же), `CHANGELOG.md` (запись Removed). Никакой архитектуры, ни миграций, ни кросс-сервисных изменений. Design-документ — формальная заглушка для удовлетворения зависимостей openspec-схемы.

## Goals / Non-Goals

**Goals:**
- Свести количество концепций фичефлагов к одной (runtime `feature_flag`), убрав parallel-no-op «compile-time» категорию.
- Удалить мёртвый код без побочных эффектов.

**Non-Goals:**
- Не вводим новые флаги, не меняем поведение существующих runtime-флагов.
- Не пытаемся восстановить «изначальный замысел» удаляемых констант — если в будущем понадобится limit для dry-run токенов или warmup-кривая, это будет отдельный change, где мы сразу подключим их к коду.

## Decisions

### Decision 1: Удалить, а не оставить на будущее

**Что:** Полностью удаляем `flags.ts`, а не оставляем «закомментированным» / помеченным TODO.

**Почему:**
- CLAUDE.md явно: «If you are certain that something is unused, you can delete it completely».
- Мёртвый код вводит в заблуждение: при чтении нового разработчика создаётся ложное впечатление, что эти флаги работают.
- История остаётся в git и в этой openspec change (после архива — `openspec/changes/archive/<date>-remove-dead-flags-ts/`), любой будущий вопрос «а где был лимит N?» отвечается одним `git log`.

### Decision 2: Не мигрировать константы в runtime feature_flag «на всякий случай»

**Что:** Не создаём в БД записи `feature_flag` под каждое удаляемое имя.

**Почему:**
- Никаких потребителей нет — запись в БД создаст ту же неподключённую декларацию, только в другом хранилище.
- runtime feature_flag система предназначена для **rollout/kill-switch активных фич**, не для «возможно когда-нибудь пригодится».

## Risks / Trade-offs

- **[Риск низкий] Кто-то на ревью спросит «а где warmup-кривая?»** → ответ: «нигде; warmup-логика ещё не реализована; когда понадобится — заведём числовой конфиг там, где он реально используется (env, БД, либо новый feature_flag с настоящим читателем)».
- **[Trade-off] Теряем декларативную «дорожную карту»** — некоторые из удаляемых имён намекали на запланированные фичи (`ENABLE_FOLLOWUP_CRON`, `WARMUP_STAGES`). Готовы принять: дорожная карта живёт в DESIGN.md/AGENTS.md, а не в коде с фейковыми константами.
