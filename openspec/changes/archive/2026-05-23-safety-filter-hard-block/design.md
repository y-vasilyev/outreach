## Context

`SafetyFilter` сейчас намеренно отдаёт LLM-вердикту только advisory-роль (`allow=true` всегда из LLM-ветки); это решение из исходного дизайна, чтобы LLM не отбрасывал валидные варианты по стилю. Hard-block ветка существует, но узкая: длина, ссылки, decline.

agency_sourcing pipeline опирается на LLM-композитор (`AgencyOpeningComposer`), у которого в промпте чётко прописаны запреты (гарантии, давление, выдуманные детали). LLM в большинстве случаев их соблюдает, но не строго — особенно мелкие модели или температуры > 0. На пилоте до 5–10 блогеров мы хотим иметь детерминистическую страховку: если запрещённый паттерн всё-таки появился в черновике, мы его НЕ отправляем.

Источник истины — `campaign_type.safetyProfile`. Сейчас он содержит `forbidden_topics`/`allowed_topics` (advisory для LLM), `max_length`, `allow_links`. Естественно добавить туда новое поле `hard_block_patterns` — список явных regex с человекочитаемой `reason`.

## Goals / Non-Goals

**Goals:**
- Дать оператору гарантию: для типов кампаний, где это критично, варианты с явным запретным паттерном не уходят в чат.
- Не сломать существующее поведение `custdev` (пустой список → старое поведение).
- Не вернуться к старой ошибке «substring matching по вибу» — паттерны явные, opt-in, в БД.
- Покрыть тестами happy path + edge cases.

**Non-Goals:**
- Не вводим UI-builder для hard-block-паттернов на этом шаге (admin может править через PATCH api либо seed). Builder задействуем когда понадобится партнёрам.
- Не вводим catch-all «правила безопасности» поверх campaign type — все hard-block per-type, иначе расходимся с уже принятым решением «safetyProfile из type, не глобальный».
- Не пытаемся валидировать regex на «правильность смысла» (это работа администратора/builder'а); только синтаксическую корректность и длину.
- Не делаем LLM hard-block (отдельный путь). Это всё ещё advisory-LLM + deterministic hard-block. Сценарии под нечёткий LLM-block (например, «эта формулировка эмоционально давящая, но не подпадает под regex») остаются темой будущего change'а.

## Decisions

### Decision 1: Регексы, не keyword-substring

**Что:** `hard_block_patterns` — массив объектов `{ id, pattern, reason, flags? }`. `pattern` компилируется в `RegExp` с allowlist флагов (`i|m|u`).

**Почему:**
- Простые подстроки слишком жёстки: «гарантирую» / «гарант-» / «100%» — нужны морфологические варианты, без регэкспа это бесконечный keyword-список.
- Регекс позволяет описать «`гарант\w*`» / «(только сегодня|осталось\s+\d+\s+мест)» / «(оплат|перевод|банковск)» одной строкой каждое.
- Регекс контролируем admin'ом через seed/builder.

**Альтернативы:**
- *LLM-вердикт с правом блока*. Отвергнуто — старая боль «LLM кладёт всё подряд» и непредсказуемое поведение мелких моделей.
- *Keyword-список из 100+ строк*. Хуже maintain'ится, тот же выразительный потенциал.

### Decision 2: Hard-block ВПЕРЁД LLM, не после

**Что:** Проверка `hard_block_patterns` идёт сразу после существующих hard-причин (`max_length`, `link`, `declined`), до `invokeJson(...)`.

**Почему:**
- Семантически: hard-block — это «никогда нельзя», не «может быть допустимо в контексте». Не должен зависеть от LLM-runtime.
- Стоимость: не тратим токены на LLM-call для заведомо отбракованного варианта.
- Тестируемость: hard-block тестируется чистым unit-тестом без LLM-моков.

### Decision 3: Идемпотентный formatting reason'ов

**Что:** `reasons[]` форматируется как `<id>:<short_reason>` (например, `agency_guarantee_promise:promise of guaranteed result`). `rewrite_hint` — человеческая строка из `pattern.reason`.

**Почему:**
- Позволяет downstream-консьюмерам (логи, метрики, UI-причина в админке) одинаково парсить.
- Совместимо с уже существующим форматом `reasons` (там сейчас тоже плоские строки вроде `link_not_allowed`).

### Decision 4: Safe-parse и fallback на malformed regex

**Что:** В `resolveSafetyContext` каждый `pattern` валидируется: сначала per-item `HardBlockPatternZ.safeParse(...)`, потом `new RegExp(...)` в `try/catch`. Сломанные silent-skip'аются (без log warn — в shared-слое нет logger'а, а добавлять зависимость только под это слишком). Длина паттерна — cap 200 chars. Флаги — allowlist `i|m|u`. SafetyFilter, получая входные данные напрямую (тесты, ad-hoc вызов), сам валидирует их через `HardBlockPatternZ` в schema; ошибки compile отдельных паттернов SafetyFilter тоже silent-skip'ает (defense in depth).

**Почему:**
- Если админ через PATCH запихнул сломанный regex, не падать всем pipeline'ом.
- Cap длины — простой ReDoS-floor (Node-RegExp не имеет timeout); 200 chars — достаточно для разумной грамматики.
- Allowlist флагов — отсечь `g` (бесмысленно для test, опасно для exec в loop), `s`/`y` (не нужны).

### Decision 5: Никаких глобальных дефолтов

**Что:** `LEGACY_SAFETY_CONTEXT.hard_block_patterns = []`. Без типа или без `safetyProfile` — нет hard-block.

**Почему:**
- Совместимо с principle «safety per-type, не глобально».
- Любое явное правило безопасности должно быть видно в config типа и audit-логе. Невидимая глобальная блокировка хуже, чем явное правило в seed.

## Risks / Trade-offs

- **[Риск] Админ написал слишком широкий regex** — например, `гарант` (без слова-границы) сматчится с «гарантия конфиденциальности» в подписи бота.
  → **Митигация:** seed-паттерны делаем со словосочетаниями, не словами-фрагментами (`гарантиру|гарантия результата|+\d+`). В docstring `SafetyProfileZ` подсказка для админов. На builder-этапе (отдельный future) добавим dry-run против корпуса драфтов.
- **[Риск] Hard-block перебарщивает, оператор не видит вариант** — все варианты могут отвалиться, диалог в `assisted` без suggestion.
  → **Митигация:** существующий код (`campaign-dispatcher.ts:317-323`) уже отслеживает лучший вариант после safety-loop; если ни один не прошёл — диалог уходит оператору ровно как при advisory-fail (известный путь). Дополнительная гарантия: тест на «все варианты заблокированы».
- **[Trade-off] Regex'ы тяжело покрыть тестами для всех естественных морфологических форм.** Принимаем — это страховка, не первичный gate. Главный фильтр — LLM-композер; hard-block ловит только то, что LLM пропустил.
- **[Trade-off] Семантика `allow=false` теперь шире** — раньше блокировало только за длину/ссылку/decline; теперь и за тематические правила. Для downstream-консьюмеров это незаметно: они уже фильтруют по `allow`.

## Migration Plan

1. Code: расширить `SafetyProfileZ` + `ResolvedSafetyContext`, добавить `hardBlockMatches()` в `SafetyFilter`, обновить тесты, обновить worker queue-пайплайны.
2. Seed: добавить `hard_block_patterns` в `agency_sourcing` seed; `custdev` остаётся `[]`.
3. Регрессия: `pnpm typecheck && pnpm lint && pnpm test`.
4. Codex review.
5. Архив openspec.
6. Коммит.

**Rollback:**
- Отдельной миграции БД здесь нет (изменения в JSON `safetyProfile`); если что — откатываем коммит и seed.
- На уже задеплоенных средах безопасно: пустой `hard_block_patterns` → старое поведение.

## Open Questions

- Нужен ли отдельный аудит-лог для каждого hard-block-матча (`AuditLog` уже есть, можно класть событие)? — Решение: пока нет; existing logging в `pino` достаточно для пилота. Если объёмы вырастут — добавим в follow-up.
- Builder-side validation regex'а (UI proportionate dry-run-pad) — Будет нужна, когда builder соберём для партнёров. Сейчас за `--strict` валидацию отвечает Zod safe-parse в `resolveSafetyContext`.
